# V2/V3 开发计划

## 目标
实现六类可重复触发的故障场景，以及日志、指标、链路、发布和事故查询接口；不实现 Agent 和复杂前端。

## 阶段
- [x] 阶段 1：核对 V1 模型与项目结构
- [x] 阶段 2：补充查询与场景响应模型
- [x] 阶段 3：实现六类故障场景造数服务
- [x] 阶段 4：实现 scenarios 和证据查询 API
- [x] 阶段 5：执行初始化、六场景和查询接口端到端验证

## 约束与决策
- 使用现有 SQLAlchemy 模型，不引入 Agent 或前端变更。
- 每次场景调用创建独立 trace_id，并在日志、链路和响应中关联。
- 场景写入在单个数据库事务中完成。
- 查询结果默认按业务时间倒序返回。
- 场景执行前校验基础服务数据；未初始化时返回 HTTP 409。

| `rg.exe` 在目标目录执行时报 Access denied | 1 | 后续文件枚举改用 PowerShell 原生命令，代码搜索按需使用 `Select-String` |
| V3 组合补丁上下文匹配失败 | 1-2 | 拆分计划、schema 和 API 补丁，减少跨文件耦合 |

## 验证结果
- FastAPI 成功注册 16 个 `/api` 路由。
- 六个场景均通过真实 HTTP 调用创建数据并返回关键证据。
- 日志关键词、指标筛选、链路详情、发布列表和事故详情查询通过。
- 慢 SQL 最大 span、连接池高使用率和关键词证据通过数据库定向校验。

## V3 目标
实现可审计的 Agent 工具层和确定性 Mock 诊断工作流，不接入真实大模型。

## V3 阶段
- [x] 阶段 6：核对 Agent、工具调用和审批模型
- [x] 阶段 7：实现工具层及统一调用审计
- [x] 阶段 8：实现 Mock 诊断规则与工作流
- [x] 阶段 9：实现会话和审批 API
- [x] 阶段 10：验证 slow-sql 诊断闭环及通用场景

## V3 决策
- 通过请求级上下文向工具注入数据库会话和 session_id，保留用户要求的工具函数签名。
- 工具输入输出使用 JSON 文本持久化，API 返回时解析为结构化数据。
- Mock Agent 只根据工具返回的 metrics、logs、traces、deployments 和 runbooks 生成报告。
- 高风险操作只创建审批单，不执行任何回滚、重启或配置变更。

## V3 验证记录
- slow-sql 场景诊断命中 inventory-service 最近发布、慢 SQL 日志、高延迟指标和最慢 span。
- 单次诊断产生 11 条成功工具调用记录。
- 输出风险等级 high，并创建 rollback_deployment 待审批单。
- 会话列表、会话详情、工具调用列表、审批列表、批准和拒绝接口通过 HTTP 验证。
